# Обзор аналогов

В этом разделе проанализированы существующие решения в области
автоматизированной разметки и парсинга веб-страниц. Аналоги разделены на
три группы по принципу работы.

---

## 1. Классические парсеры по жёстким правилам

### 1.1 Scrapy

[scrapy.org](https://scrapy.org) — наиболее популярный open-source
фреймворк для веб-скрапинга на Python.

**Сильные стороны.** Зрелая экосистема, асинхронная архитектура на Twisted,
встроенная поддержка пайплайнов обработки, middleware, экспорта в JSON/CSV.
Огромное community и тысячи готовых интеграций.

**Слабые стороны.** Требует явного описания XPath/CSS-селекторов для каждого
сайта. При любом редизайне (переименование класса, перестановка блоков)
spider перестаёт работать — нужна ручная отладка. Не справляется со страницами
на React/Vue без отдельной интеграции со Splash или Playwright.

**Где наш проект лучше.** Не требует ручного описания селекторов — LLM
сама их генерирует по образцу страницы.

### 1.2 BeautifulSoup + requests / httpx

[crummy.com/software/BeautifulSoup](https://www.crummy.com/software/BeautifulSoup/) —
не фреймворк, а библиотека для парсинга HTML. Часто используется в связке с
``requests``.

**Сильные стороны.** Лёгкая, простая в освоении, гибкая.

**Слабые стороны.** Полностью «низкоуровневая»: разработчик пишет логику с
нуля для каждого сайта. Нет встроенной поддержки JavaScript-рендеринга,
ретраев, прокси, антибот-обхода.

**Где наш проект лучше.** Используем BeautifulSoup как инструмент очистки
и применения уже найденных селекторов, но **разметку строит LLM** —
человеко-часы не тратятся.

### 1.3 Готовые библиотеки под маркетплейсы

Например, [wb-api-python](https://github.com/wildberries-tech/wb-python),
[bs4-ozon-parser](https://pypi.org/project/ozon-parser/) и подобные.

**Сильные стороны.** Заточены под один сайт, быстрые и проверенные.

**Слабые стороны.** Работают только на одном сайте, при изменении дизайна
ломаются. Не покрывают остальной рынок.

**Где наш проект лучше.** Универсальный — работает на любом e-commerce
сайте, не только на маркетплейсах.

## 2. No-code платформы

### 2.1 Octoparse

[octoparse.com](https://www.octoparse.com) — десктоп-приложение со
встроенным браузером для визуального построения парсеров.

**Сильные стороны.** Удобный GUI: пользователь кликает на элементы страницы,
и платформа автоматически предлагает XPath/CSS. Облачное выполнение, API.

**Слабые стороны.** Платный (бесплатный тариф сильно ограничен), закрытый
исходный код. При смене дизайна сайта пользователь всё равно вручную
перестраивает шаги. Нет автоматической миграции на новый макет.

**Где наш проект лучше.** Бесплатный, open-source. Селекторы переопределяются
одним обращением к LLM, без ручного клика по элементам. Конфиг хранится
в открытом формате TOON, его можно отредактировать в любом редакторе.

### 2.2 ParseHub

[parsehub.com](https://www.parsehub.com) — конкурент Octoparse с похожим UX.

Те же сильные и слабые стороны. Дополнительный минус — отсутствие нормального
self-hosted режима.

### 2.3 Apify

[apify.com](https://apify.com) — облачная платформа с готовыми «акторами»
для парсинга популярных сайтов.

**Сильные стороны.** Маркетплейс готовых решений (десятки тысяч «акторов»).
Хорошее качество скрапинга, прокси из коробки.

**Слабые стороны.** Платное (модель оплаты — за compute units). Закрытая
экосистема: акторы — это контейнеры на их инфраструктуре, не получится
запустить локально или в своём облаке.

**Где наш проект лучше.** Self-hosted, прозрачный, без vendor lock-in.

## 3. AI/LLM-based решения

### 3.1 Diffbot

[diffbot.com](https://www.diffbot.com) — коммерческий сервис, который
использует computer vision и ML для автоматического извлечения данных.

**Сильные стороны.** Очень высокое качество извлечения, поддерживает не
только товары, но и статьи, видео, профили. Огромная база уже размеченных
сайтов.

**Слабые стороны.** Платный (минимальный тариф $299/мес). Closed-source.
Не даёт пользователю CSS-селекторы — только готовые JSON-результаты, что
неудобно для downstream-обработки (например, передать в свой парсер).

**Где наш проект лучше.** Open-source. Возвращает **сами CSS-селекторы**,
которые можно использовать вне нашего сервиса (например, в Scrapy). Дешевле:
основные расходы — бесплатный HuggingFace Inference или копейки за
GPT-4o-mini.

### 3.2 ScrapingBee + AI Extraction

[scrapingbee.com](https://www.scrapingbee.com) — облачный browser-as-a-service
для парсинга с возможностью передать prompt и получить JSON.

**Сильные стороны.** Простой API, прокси/captcha solving в комплекте.

**Слабые стороны.** Платный. AI-извлечение — это просто «передайте промпт»,
сами селекторы не возвращаются: каждый запрос — это новый вызов LLM, что
дороже нашего подхода с кэшированием селекторов.

**Где наш проект лучше.** Бесплатный и self-hosted. Селекторы один раз
генерируются и переиспользуются: парсинг тысячи страниц одного домена —
один вызов LLM плюс 1000 быстрых BS4-запросов.

### 3.3 GPT-4 для скрапинга «руками»

Многие энтузиасты используют ChatGPT напрямую: просто копируют HTML и
просят достать данные. Это паттерн, который наш сервис автоматизирует и
ставит на промышленные рельсы.

**Слабые стороны паттерна.** Нет автоматизации, нет валидации, нет
сохранения селекторов, нет UI. Не масштабируется.

**Где наш проект лучше.** Полностью автоматизированный пайплайн с
валидацией, БД, фоновыми задачами (через Celery в будущей итерации),
двумя режимами работы (генерация селекторов или прямое извлечение).

## Вывод

Ни одно из изученных решений не сочетает **одновременно**:

1. Использование LLM для автоматической разметки.
2. Возврат **самих CSS-селекторов** (а не только данных), что позволяет
   переиспользовать их вне сервиса.
3. Open-source + self-hosted.
4. Поддержку SPA через headless-браузер.

LLM Web Parser закрывает именно эту нишу. Кэширование селекторов
обеспечивает скорость и стоимость на уровне классических парсеров, а
автоматическая генерация — гибкость и устойчивость к редизайну, типичные
для AI-решений.
