## Принципы проектирования

### 1. Чёткое разделение ответственности по слоям

Каждый модуль решает ровно одну задачу:

| Модуль | Ответственность |
|--------|-----------------|
| `app/scraper/fetcher.py` | Загрузить HTML через Playwright |
| `app/scraper/cleaner.py` | Убрать мусор, ужать HTML до 12K символов |
| `app/llm/parser.py` | Вызвать LLM и извлечь JSON из её ответа |
| `app/scraper/applier.py` | Применить готовые CSS-селекторы к HTML |
| `app/models/schemas.py` | Pydantic-валидация всех data-классов |
| `app/database/db.py` | Async-CRUD к SQLite |
| `app/toon/serializer.py` | Сериализация в формат TOON |
| `app/api/routes.py` | HTTP-эндпоинты, оркестрация остальных модулей |

Это позволяет тестировать каждый модуль изолированно (см. `tests/`).

### 2. Lazy initialization для LLM

`LLMParser()` создаётся при старте FastAPI, но реальное соединение с
HuggingFace API устанавливается только при первом фактическом запросе.
Это важно для тестов: можно создать `LLMParser` со stub-ключом и
подменить `_chain` через `unittest.mock` — без попытки реально
дёрнуть API.

### 3. Fallback внутри LLM-слоя

LangChain активно ломается с каждой минорной версией. Чтобы курсовая не
зависела от его капризов, реализован двухуровневый fallback:

1. **Уровень 1**: цепочка `template | llm` через LangChain.
2. **Уровень 2**: при любой ошибке цепочки — прямой вызов
   `huggingface_hub.InferenceClient.chat_completion()`.

В тестах оба уровня замокированы (`patch("...._chain")`,
`patch("...._direct_hf_call")`).

### 4. Толерантный JSON-парсер

Открытые модели регулярно возвращают:

* JSON с trailing comma;
* JSON в markdown-блоке;
* JSON с одинарными кавычками вместо двойных;
* JSON с пояснительным текстом до и после;
* JSON с Python-стилем (`None`, `True`, `False`).

Парсер `_extract_json` сначала пробует `json.loads` напрямую, потом
итеративно чинит каждую из перечисленных проблем. Покрывается 6 тестами
в `tests/test_llm_parser.py`.

### 5. Pydantic как контракт между слоями

Все межслойные интерфейсы — это Pydantic-модели:

* `ParseRequest` — что приходит от клиента;
* `ProductData` — что приходит из LLM (после валидации);
* `DomainMarkup` / `ProductPageSelectors` — формат для TOON;
* `ParseResponse` — что возвращается клиенту.

Каждая модель содержит `field_validator`'ы: цена должна быть
положительной и реалистичной (`< 10 млн`), URL изображения — http(s),
URL запроса — http(s), длина — не более 2048 символов.

### 6. Универсальность

В отличие от исходной задачи «парсер Wildberries», сервис **не
привязан ни к одному маркетплейсу**:

* `source` по умолчанию = `other`;
* `dashboard.html` использует placeholder с примером в стиле
  `example.com`;
* `prompts/templates.py` подставляет `source` в промпт только для
  контекста, не как фильтр.

Это позволяет обрабатывать **любой** e-commerce сайт.

### 7. Кэширование через CSS-селекторы

Главное архитектурное решение — кэшировать **CSS-селекторы**, а не
сырые результаты парсинга. Преимущества:

* Один вызов LLM покрывает любое число страниц одного домена.
* При изменении содержимого товара (цена обновилась, скидка появилась)
  не требуется повторный вызов LLM — селектор найдёт новое значение.
* При редизайне сайта достаточно повторно вызвать
  `POST /api/selectors/generate` — никаких других изменений в коде.
* TOON-конфиг можно отдать другим инструментам (Scrapy, Octoparse) — это
  открытый формат.

## Поток данных: режим CSS-селекторов

```
POST /api/selectors/generate
  ▼
1. fetch_page(url)              # Playwright → HTML
  ▼
2. clean_html(html)             # → 12K-символьный snippet
  ▼
3. llm.generate_selectors()     # LangChain → JSON селекторов
  ▼
4. validate_selectors()         # каждый селектор проверяется на HTML
  │                             #   ничего не нашёл → null + комментарий
  ▼
5. DomainMarkup(...)            # Pydantic
  ▼
6. save_domain_markup()         # SQLite UPSERT по домену
  ▼
7. response: { domain: {...} }


POST /api/selectors/apply       # БЕЗ LLM
  ▼
1. get_domain_markup(domain)    # SQLite SELECT
  ▼
2. fetch_page(url)              # Playwright → HTML
  ▼
3. apply_selectors(html, sel)   # BeautifulSoup, без LLM
  │                             #   нормализация цены и артикула
  ▼
4. ProductData(...)             # Pydantic
  ▼
5. save_parse_result()          # SQLite INSERT
  ▼
6. response: { product: {...} }
```

## Что НЕ реализовано

Намеренно опущено для соблюдения сроков и принципа «лучше меньше, да
лучше»:

| Не сделано | Почему |
|-----------|--------|
| **Redis-кэш** | SQLite-таблица `domain_markups` уже кэширует селекторы (≈ ТТЛ ∞). Для курсового достаточно. |
| **Celery + workers** | Запросы синхронные, время ≈ 5–15 сек, что приемлемо для дашборда. Async через FastAPI достаточно. |
| **PostgreSQL** | SQLite справляется при одном инстансе. Миграция — это поменять connection string и драйвер; интерфейс `app/database/db.py` готов. |
| **Auth / JWT** | Сервис локальный/демонстрационный. Auth-логику легко добавить через `Depends`. |
| **Парсинг карточек корзины и заказа** | Модели `CartPageSelectors` и `OrderPageSelectors` определены, но генерация LLM пока покрывает только `productPage`. |

## Решения, принятые сознательно

### Почему FastAPI, а не Flask

FastAPI даёт из коробки:
* автогенерируемую OpenAPI документацию (`/docs`);
* нативный `async/await`;
* интеграцию с Pydantic (валидация на уровне сигнатуры endpoint).

Для I/O-bound сервиса (загрузка страниц + ожидание LLM) асинхронная
архитектура критична.

### Почему SQLite, а не PostgreSQL

* Один файл, нулевая инфраструктура.
* aiosqlite даёт async-интерфейс без накладных расходов.
* Курсовой проект, демо-сценарий — единичные пользователи.

При необходимости масштабирования замена на PostgreSQL займёт ≤ 1 часа
(тот же SQLAlchemy-стиль API).

### Почему Llama-3.1-8B, а не GPT-4

* Бесплатно через HuggingFace Inference.
* Достаточно для извлечения структурированных данных из HTML.
* Если потребуется качество — переключение на OpenAI делается через
  переменную окружения `OPENAI_API_KEY` без правки кода.

### Почему TOON, а не JSON / YAML

Так задано в ТЗ Parsmart (`domains.toon` — пример из задания).
Соответствие конкретной структуре было одним из требований.
